# yaritai-dev：開発の進め方

## このスキルの目的
kanamiさん（UIUXデザイナー、開発未経験）のiOSアプリ開発を、どのセッション・どのモデルでも同じ品質でサポートし続けるための手順書。過去の経緯を知らなくても、これを読めば続きから再開できる。

## 最初にやること（毎回必ず）
1. `CLAUDE.md` を読む（話し方のルールと決定事項がある。特に「ユーザーとの話し方」は厳守）
2. `docs/開発ノート.md` を読む（現在の状況と次のタスクが書いてある）
3. kanamiさんに「いま〇〇まで進んでいます。今日は△△をやりましょうか」と現在地を短く共有してから始める

## 開発フェーズと進め方

### フェーズ1：HTMLモック（prototype/index.html）
- ブラウザで動くモック。保存フローの手数・リマインドの文言・完了体験をここで固める
- 修正依頼が来たらこのファイルを直し、`open prototype/index.html` でブラウザ確認してもらう
- デザイン（色・余白・アニメーション・文言）はkanamiさんの専門。実装の制約だけ伝えて判断は委ねる

### フェーズ2：iOS版（SwiftUI）
- Xcodeが必要。未インストールなら App Store から「Xcode」をインストールしてもらう（巨大なので先にダウンロード開始を促す）
- プロジェクトは `ios/` ディレクトリに。構成の方針：
  - SwiftUI、外部ライブラリなし。データ保存はSwiftData
  - `ScrapRepository`（データの窓口役）を1枚挟んで、保存先やAI呼び出し先を将来差し替え可能にする
  - 共有シート対応：Share Extension ＋ App Groups。**着手前にkanamiさんへ「アプリ本体に付ける小さな受付窓口」とたとえて説明する**
  - AI日付読み取り：Claude APIを使う方針だが、**API鍵をアプリに直接埋め込まない**こと（鍵が盗まれ請求が来る）。着手時に「小さな中継サーバーを置く／読み取り機能の課金設計」をkanamiさんと決める（`docs/企画書.md` の「まだ決めていないこと」参照）
  - リマインドはローカル通知（UserNotifications）。開始3日前／終了1週間前／日付なしは定期（例：1ヶ月後）
  - 画面：ホーム一覧 / 保存・編集シート / 詳細 の3つから始める
- HTMLモックで確定した見た目・動きを忠実に移植する

### フェーズ3：リリース
- `.claude/skills/yaritai-release/SKILL.md` を読んでその手順に従う

### フェーズ4：改善サイクル
- リリース後の要望・改善は `docs/企画書.md` の「まだ決めていないこと」「v2以降」も参照しながら小さく出す

## 作業の終わり方（毎回必ず）
`docs/開発ノート.md` を更新する：
- 「現在の状況」を今日やったことを反映した内容に書き換える
- 「次にやること」を具体的に書く（次のセッションのClaudeがそのまま着手できる粒度で）
- 日付入りの作業ログを1行追記する

## 説明のしかたの例
- ビルド → 「コードを組み立てて、動くアプリの形にすること」
- シミュレータ → 「Macの画面の中に出てくる仮想のiPhone」
- Share Extension → 「アプリ本体に付ける、共有シート専用の小さな受付窓口」
- App Groups → 「アプリ本体と受付窓口が同じ引き出しを共有するための設定」
- ローカル通知 → 「ネット不要で、iPhone自身が時間になったら鳴らす通知」
- API → 「よそのサービスに仕事を頼むための注文窓口」
- リポジトリ／Repository → 「データの窓口役。中身が端末内でもネットでも、アプリ側は同じ聞き方で済む」
- Git・コミット → 「作業のセーブポイント。いつでも巻き戻せる記録」
このように、新しい用語が出るたび一言のたとえを添える。
